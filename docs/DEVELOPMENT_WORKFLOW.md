# GD Trade 多 Agent 开发流程

## 标准流程

    产品需求
       ↓
    GD Architect 设计
       ↓
    GD Developer 实现
       ↓
    GD QA 测试
       ↓
    代码合并

## 1. 产品需求

- 明确用户目标、版本范围、验收条件和禁止事项。
- 在 TASK_QUEUE.md 中登记优先级、状态、负责角色和预期文件范围。
- 涉及行情、风险或交易辅助时，明确数据时效、风险否决和人工确认边界。

产出：已登记的任务、验收条件和范围说明。

## 2. GD Architect 设计

- 阅读路线图、当前架构、任务队列和上一任务完成记录。
- 评估模块边界、接口影响、数据迁移、并行冲突和测试策略。
- 冻结本次迭代需要共享的 Repository 接口、领域模型和 UI 状态契约。
- 为各 Agent 指定分支、目录或文件所有权。

产出：技术方案、接口契约、任务拆分和合并顺序。

## 3. GD Developer 实现

- 在独立分支或 worktree 中实现已确认方案。
- 只修改分配范围内文件，不顺带重构无关模块。
- UI 通过 ViewModel 和 StateFlow 获取状态，不直接访问 DAO。
- 业务规则放入 UseCase 或 Domain Logic，数据访问通过 Repository。
- 完成局部单元测试和编译检查。

产出：实现代码、测试代码和必要文档更新。

## 4. GD QA 测试

- 根据验收条件执行单元测试、迁移测试、编译检查和回归检查。
- 检查静态行情标识、风险否决优先级和非自动交易边界。
- Bug 报告必须包含复现步骤、预期结果、实际结果和影响范围。
- 未经 GD Architect Agent 确认，不重构公共接口或核心架构。

产出：测试结果、Bug 清单、性能观察和剩余风险。

## 5. 代码合并

- 按架构与接口、Data、Domain、UI、测试与文档的顺序合并。
- 每次合并前同步目标分支，解决冲突后重新运行相关检查。
- 由 GD Architect Agent 核对接口与文档，由 GD QA Agent 核对最终测试结果。
- 更新 TASK_COMPLETION.md，确保六项交接信息完整。

产出：可追溯提交、完整交接记录和更新后的任务状态。

## 并行开发约束

- 可以并行：不同目录且不改变共享契约的任务，例如独立 UI 页面、独立 Domain 规则、独立测试和文档。
- 需要串行：Repository 接口、领域模型、Room Schema、导航契约和共享 UI 状态变更。
- 共享文件由单一 Agent 主写，其他 Agent 通过评审意见或独立补丁协作。
- 出现接口冲突时暂停实现，由 GD Architect Agent 重新确认契约和合并顺序。

## 任务状态

任务状态统一使用：

- 待开始。
- 进行中。
- 待测试。
- 待合并。
- 完成。
- 阻塞。

状态为“阻塞”时必须记录阻塞原因、已尝试措施和解除条件。

## 最小交接清单

- 完成内容。
- 修改文件。
- 架构影响。
- 测试结果。
- 已知问题。
- 下一步建议。
- 当前分支和提交号。
- 是否存在未提交或未合并改动。
