# V1.1 真实设备覆盖升级测试方案

## 目标

验证 GD Trade 从使用 SharedPreferences 的旧版本原位升级到 Room 版本后，`Position`、`TradeRecord`、`StockCandidate` 数据完整保留，并且重复启动或迁移过程被中断时不会产生重复数据或数据丢失。

## 当前执行状态

- 测试设备：OPPO PLG110，Android 16，设备连接正常。
- 当前已安装版本：`versionName 1.5.0`、`versionCode 2`。
- 当前设备已经是 Room 新版本，无法从现状反向证明旧版到新版的原位迁移。
- 本轮未卸载应用、未清除应用数据、未降级安装，避免破坏用户现有持仓和交易记录。
- 结论：测试方案已建立，真实旧版数据覆盖升级仍待受控执行。

## 前置条件

1. 准备使用 SharedPreferences 存储数据的旧版 APK，建议固定为 `versionCode 1`。
2. 准备本轮已构建的新版本 APK：`app/build/outputs/apk/debug/app-debug.apk`。
3. 使用测试设备或已完成完整备份的设备，不直接使用唯一生产数据副本。
4. 确认 ADB 已识别设备，并记录设备序列号、Android 版本和应用版本。
5. 升级过程中不得执行卸载应用或 `pm clear`，否则无法验证原位数据迁移。

## 旧版数据基线

在旧版中建立并截图记录以下数据，字段值应具有唯一性，便于升级后逐项核对：

| 类型 | 最低基线 | 必查字段 |
| --- | --- | --- |
| Position | 3 条，其中一条使用非默认数量和备注 | 代码、名称、数量、备注、显示顺序 |
| TradeRecord | 3 条，其中 2 条内容完全相同 | 日期、代码、名称、方向、价格、数量、备注、重复条数 |
| StockCandidate | 3 条，其中一条 `riskDeniedBuy=true` | 代码、名称、主题、原因、信号状态、风险否决状态、显示顺序 |

建议额外记录三类数据的总数，并导出或截图旧版界面作为验收证据。

## 正常迁移步骤

1. 安装旧版 APK 并建立上述数据基线。
2. 强制停止旧版应用，确认旧版数据重新启动后仍存在。
3. 执行原位覆盖安装：`adb install -r <新版 APK 路径>`。
4. 首次启动新版，等待首页数据加载完成。
5. 逐条核对三类数据的数量与全部字段。
6. 确认重复的 TradeRecord 仍为 2 条，未被意外合并。
7. 强制停止并重新启动应用两次，确认数据不重复、不回退。
8. 新增、修改或删除一条测试数据，再次重启，确认后续写入由 Room 持久化。

## 空数据迁移步骤

1. 在旧版中将 Position、TradeRecord、StockCandidate 全部清空。
2. 原位覆盖安装新版。
3. 首次和第二次启动后均应保持为空，不得自动恢复默认样例数据。

## 重复启动步骤

1. 完成正常迁移后记录三类数据数量。
2. 连续执行三次“强制停止并重新启动”。
3. 每次启动后核对数量和字段，任何记录不得重复导入或被旧数据覆盖。

## 异常中断步骤

1. 在测试设备恢复旧版数据基线后，覆盖安装新版。
2. 首次启动新版后立即执行 `adb shell am force-stop com.gudian.gdtrade`，模拟迁移启动阶段被中断。
3. 再次启动应用，核对三类数据完整性和迁移结果。
4. 连续重启两次，确认重试不会产生重复数据。
5. 若出现数据缺失，保留应用数据、logcat 和数据库文件，不继续覆盖操作。

说明：应用当前使用单个 Room 事务写入三类数据。自动化测试已验证事务异常会回滚并可重试；真机步骤用于覆盖进程终止、文件系统和厂商系统行为。

## 建议采集证据

- 升级前和升级后的应用版本信息。
- 三类数据升级前后截图及数量对照表。
- 首次启动、异常中断和重启后的 logcat。
- Debug 构建可在设备允许时使用 `run-as com.gudian.gdtrade` 检查数据库和 SharedPreferences 标记，但不得修改文件。

## 通过标准

- Position、TradeRecord、StockCandidate 的数量和字段全部一致。
- 重复 TradeRecord 完整保留。
- 空数据升级后仍为空。
- 重复启动不重复导入、不覆盖 Room 新数据。
- 异常中断后可恢复，最终不存在部分提交。
- 应用无崩溃，首页状态正常组合，风险引擎仍可否决买入研究信号。

## 失败处理

- 任何一类数据缺失、重复或字段变化均判定失败。
- 保留失败设备现场，禁止卸载或清除数据。
- 记录旧版 APK、新版 APK、设备系统、复现步骤和日志后交由 GD Developer Agent 修复。
